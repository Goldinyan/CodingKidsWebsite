Ja, absolut! Das ist genau der richtige Architektur-Schritt, wenn eine App wächst. Man nennt das einen **Global Data Provider** oder **AppState Context**.

Wenn du das alles direkt in den `AuthContext` klatschst, wird dieser allerdings schnell zu einem riesigen "God-Object", das schwer zu warten ist. Die sauberste Lösung ist, die Daten-Zustände (Mentoren, Events, Kurse) in **einem eigenen State-Context** zu bündeln, der sich aber automatisch an den `AuthContext` hängt.

Hier ist das Blueprint, wie du einen kombinierten `DataContext` baust, der alles effizient verwaltet:

### 1. Der kombinierte `DataContext.tsx`

Erstell eine neue Datei, z. B. `@/context/DataContext.tsx`. Dieser Provider lauscht auf den `useAuth()`. Sobald der User eingeloggt ist, lädt er **einmal parallel** alle benötigten Daten (Mentoren, Events, Kurse) herunter.

```tsx
"use client";

import React, { createContext, useContext, useState, useEffect, useRef } from "react";
import { useAuth } from "@/BackEnd/AuthContext";
import { getAllMentors, getAllEvents, getAllCourses } from "@/lib/db"; // Deine DB-Funktionen
import type { Mentor, Event, Course } from "@/BackEnd/type";

interface DataContextType {
  mentors: Mentor[];
  events: Event[];
  courses: Course[];
  loadingData: boolean;
  refreshData: (type?: "all" | "mentors" | "events" | "courses") => Promise<void>;
}

const DataContext = createContext<DataContextType | undefined>(undefined);

export function DataProvider({ children }: { children: React.ReactNode }) {
  const { user, userRole, loading: authLoading } = useAuth();
  
  const [mentors, setMentors] = useState<Mentor[]>([]);
  const [events, setEvents] = useState<Event[]>([]);
  const [courses, setCourses] = useState<Course[]>([]);
  const [loadingData, setLoadingData] = useState<boolean>(true);

  // Verhindert mehrfaches Fetchen bei identischem Auth-Status
  const hasFetchedFor = useRef<string | null>(null);

  const fetchData = async (target: "all" | "mentors" | "events" | "courses" = "all") => {
    const currentKey = user ? `${user.uid}-${userRole}` : "guest";
    
    setLoadingData(true);
    try {
      // ⚡ Parallel-Fetching mittels Promise.all für maximale Performance
      const promises: Promise<any>[] = [];
      
      const shouldFetchMentors = target === "all" || target === "mentors";
      const shouldFetchEvents = target === "all" || target === "events";
      const shouldFetchCourses = target === "all" || target === "courses";

      if (shouldFetchMentors) promises.push(getAllMentors(user?.uid, userRole));
      if (shouldFetchEvents) promises.push(getAllEvents()); // Beispiel-Funktion
      if (shouldFetchCourses) promises.push(getAllCourses()); // Beispiel-Funktion

      const results = await Promise.all(promises);
      
      let index = 0;
      if (shouldFetchMentors) {
        setMentors(results[index++].sort((a: any, b: any) => a.id - b.id));
      }
      if (shouldFetchEvents) {
        setEvents(results[index++]);
      }
      if (shouldFetchCourses) {
        setCourses(results[index++]);
      }

      if (target === "all") {
        hasFetchedFor.current = currentKey;
      }
    } catch (error) {
      console.error("Fehler beim Laden des App-Zustands:", error);
    } finally {
      setLoadingData(false);
    }
  };

  // Synchronisiert die Daten mit dem AuthContext
  useEffect(() => {
    if (authLoading) return;

    const currentKey = user ? `${user.uid}-${userRole}` : "guest";
    // Nur laden, wenn sich die Session-ID geändert hat
    if (hasFetchedFor.current !== currentKey) {
      fetchData("all");
    }
  }, [authLoading, user, userRole]);

  // Einzelne Refresh-Funktion, um gezielt nach Mutationen (z.B. Event erstellt) zu updaten
  const refreshData = async (target: "all" | "mentors" | "events" | "courses" = "all") => {
    await fetchData(target);
  };

  return (
    <DataContext.Provider value={{ mentors, events, courses, loadingData, refreshData }}>
      {children}
    </DataContext.Provider>
  );
}

export function useAppData() {
  const context = useContext(DataContext);
  if (!context) throw new Error("useAppData muss innerhalb eines DataProviders verwendet werden");
  return context;
}

```

### 2. Provider in der `layout.tsx` verschachteln

Da der `DataProvider` den `useAuth()` Hook aufruft, muss er **innerhalb** des `AuthProviders` liegen. Die Reihenfolge im Root-Layout sieht dann so aus:

```tsx
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <ThemeProvider>
      <AuthProvider>
        <DataProvider>
          {children}
        </DataProvider>
      </AuthProvider>
    </ThemeProvider>
  );
}

```

### 3. Die Nutzung in deinen Views

Jetzt kannst du von *jeder* beliebigen Komponente aus extrem sauber auf alle Daten zugreifen. Keine unnötigen API-Calls mehr, keine doppelten Schleifen:

#### In der Mentoren-Ansicht:

```tsx
const { mentors, loadingData } = useAppData();
// ... useMemo zum Filtern/Sortieren nutzen

```

#### In einer Kurs-Übersicht oder im Dashboard:

```tsx
const { courses, events, refreshData } = useAppData();

const handleCreateCourse = async () => {
  await createNewCourse(courseData);
  // Re-fetcht NUR die Kurse aus Firebase, um Traffic zu sparen!
  refreshData("courses"); 
};

```

### Warum diese Architektur genial ist:

1. **`Promise.all` Effizienz:** Anstatt dass Mentoren, Events und Kurse nacheinander tröpfeln und die App dreimal ruckeln lassen, werden alle Daten gleichzeitig in einem einzigen asynchronen Rutsch geladen.
2. **Selektiver Refresh:** Wenn ein User ein Event bucht oder du im Admin-Bereich einen Kurs änderst, rufst du einfach `refreshData("courses")` auf. Es werden nicht alle Daten neu geladen, sondern nur die betroffene Collection.
3. **Zentrale Wahrheit (Single Source of Truth):** Wenn ein Kurs auf der Startseite angezeigt wird und im Dashboard, zeigen beide Komponenten garantiert exakt dieselben Daten an.
